Create `src-tauri/src/git/queue.rs` implementing the queue orchestrator. Follow the research-recommended pattern from 17-RESEARCH.md exactly.

**Structs to create:**

1. `QueueActiveFlag(pub AtomicBool)` -- managed state wrapper for watcher suppression.

2. `QueueProgress` (Serialize, Clone, Debug) with fields: `index: usize`, `total: usize`, `branch: String`, `status: String` (values: "active", "complete", "failed", "rolled_back").

3. `QueueResult` (Serialize, Clone, Debug) with fields: `success: bool`, `completed: usize`, `total: usize`, `failed_branch: Option<String>`, `error: Option<String>`, `build_range: Option<(u32, u32)>`.

**Function `execute_queue()`:**

Signature:
```rust
pub fn execute_queue(
    app: &tauri::AppHandle,
    project_path: &str,
    branches: Vec<String>,
    merge_target: &str,
    build_patterns: &[BuildFileConfig],
    changelog_config: &Option<ChangelogConfig>,
    queue_flag: &QueueActiveFlag,
) -> Result<QueueResult, GitError>
```

Implementation per MERGE-04, MERGE-05, MERGE-06:

1. Set `queue_flag.0.store(true, Ordering::SeqCst)` to suppress watcher.
2. Open repo, record `snapshot_oid = repo.head()?.target()`. Drop repo immediately to avoid Windows file lock issues (per Pitfall 2 from research).
3. Call `build::detect_current_build(project_path, build_patterns)` ONCE to get base build number (per MERGE-05 -- no disk reads between merges).
4. Loop over branches:
   a. Emit "merge-queue-progress" event with status "active" via `app.emit()`.
   b. Increment `current_build`. Set `override_build = if build_patterns.is_empty() { None } else { Some(current_build) }`.
   c. Create `MergeContext::new(...)` with the override_build.
   d. Run `merge_execute(&mut ctx).and_then(|_| merge_bump(&mut ctx)).and_then(|_| merge_changelog(&mut ctx)).and_then(|_| merge_commit(&mut ctx))`.
   e. On success: increment completed, emit "complete" progress.
   f. On error: emit "failed" for current branch, then ROLLBACK.

5. **Rollback (on error):**
   a. Open repo, find commit by snapshot_oid, call `repo.reset(commit.as_object(), git2::ResetType::Hard, None)`.
   b. Emit "rolled_back" status for each previously completed branch (indices 0..completed).
   c. Clear queue flag, emit forced "git-changed" event, return QueueResult with success=false.

6. **Success path:** Clear queue flag, emit forced "git-changed" event, return QueueResult with success=true and build_range.

The "git-changed" forced refresh event must use the same JSON shape the watcher uses:
```rust
let _ = app.emit("git-changed", serde_json::json!({
    "project_path": project_path,
    "change_type": "refs_changed"
}));
```

Use `tauri::Emitter` trait import for `app.emit()`.

Add `pub mod queue;` to `src-tauri/src/git/mod.rs`.

**Critical anti-patterns to avoid (from research):**
- Do NOT hold a Repository handle during the pipeline step loop -- pipeline steps open their own.
- Do NOT read build number from disk between merges -- use override_build exclusively.
- Do NOT forget to clear the queue flag on ALL exit paths (success AND failure).

**1. Add merge_queue_execute command to git_commands.rs:**

Follow the existing `merge_branch` command pattern. The new command:

```rust
#[tauri::command]
pub fn merge_queue_execute(
    app_handle: tauri::AppHandle,
    project_path: String,
    project_name: String,
    branches: Vec<String>,
    merge_target: String,
    build_patterns: Vec<BuildFileConfig>,
    changelog_config: Option<ChangelogConfig>,
    write_lock: tauri::State<'_, Mutex<()>>,
    queue_flag: tauri::State<'_, crate::git::queue::QueueActiveFlag>,
) -> Result<crate::git::queue::QueueResult, GitError> {
    let _guard = write_lock.lock().map_err(|e| {
        GitError::MergeAborted(format!("Failed to acquire write lock: {}", e))
    })?;
    crate::git::queue::execute_queue(
        &app_handle, &project_path, branches, &merge_target,
        &build_patterns, &changelog_config, &queue_flag,
    )
}
```

Note: `project_name` is accepted for IPC symmetry with merge_branch but not used in queue logic. The `_guard` holds the Mutex for the entire queue duration, preventing concurrent merge operations.

Modify `start_watcher()` to accept and propagate the QueueActiveFlag:

**Recommended approach:** Add `queue_active: std::sync::Arc<std::sync::atomic::AtomicBool>` parameter to both `start_watcher()` and `process_events()`. This way the watcher thread and the managed state share the same AtomicBool.

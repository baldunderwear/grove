       **NotificationState struct** (managed Tauri state):
       ```rust
       use std::collections::HashMap;
       use std::sync::Mutex;

       pub struct NotificationState {
           /// branch_key ("project_name:branch_name") -> was merge-ready last check
           merge_ready: HashMap<String, bool>,
           /// branch_key -> was stale last check
           stale: HashMap<String, bool>,
       }

       impl NotificationState {
           pub fn new() -> Self {
               Self {
                   merge_ready: HashMap::new(),
                   stale: HashMap::new(),
               }
           }
       }
       ```

       **`check_and_notify(app: &tauri::AppHandle, state: &Mutex<NotificationState>)`**
       - Load config via `crate::config::persistence::load_or_create_config(app)`
       - For each project in config.projects:
         - Call `crate::git::branches::list_worktree_branches()` (catch errors, skip project on failure)
         - For each branch:
           - **Merge-ready check** (if settings.notify_merge_ready):
             - is_merge_ready = branch.ahead > 0 && !branch.is_dirty
             - key = format!("{}:{}", project.name, branch.name)
             - If is_merge_ready && !was_previously_notified: fire notification, mark as notified
             - If !is_merge_ready: reset to false (so transition fires again later)
           - **Stale check** (if settings.notify_stale_branch):
             - Calculate age: current_time - branch.last_commit_timestamp
             - is_stale = age > 7 * 24 * 3600 (7 days in seconds)
             - Same transition logic: notify only on false->true transition
       - Use `tauri_plugin_notification::NotificationExt` to fire:
         ```rust
         use tauri_plugin_notification::NotificationExt;
         let _ = app.notification()
             .builder()
             .title("Branch Ready to Merge")
             .body(&format!("{} in {} is merge-ready", branch_name, project_name))
             .show();
         ```
       - For stale notifications:
         ```rust
         .title("Stale Branch Detected")
         .body(&format!("{} in {} has no activity for 7+ days", branch_name, project_name))
         ```

       **`notify_merge_complete(app: &tauri::AppHandle, branch_name: &str, project_name: &str, build_number: &str)`**
       - Public function callable from merge command completion
       - Check settings.notify_merge_complete before firing
       - Fire notification:
         ```rust
         .title("Merge Complete")
         .body(&format!("{} merged into {} (build {})", branch_name, project_name, build_number))
         ```

    2. Update src-tauri/src/lib.rs:
       - Add `mod notifications;` and `mod fetch;` (fetch module comes in Task 2)
       - Add managed state: `.manage(Mutex::new(notifications::NotificationState::new()))`
       - In setup(), after watcher startup, call check_and_notify once for initial state:
         ```rust
         let app_handle_notify = app.handle().clone();
         let notif_state = app.state::<Mutex<notifications::NotificationState>>();
         notifications::check_and_notify(&app_handle_notify, &notif_state);
         ```

    3. Wire notifications to watcher events. In the watcher's process_events loop (src-tauri/src/watcher/mod.rs), after emitting "git-changed" events, also trigger notification checks. However, since the watcher doesn't have access to the notification state, instead emit a separate internal "check-notifications" approach:
       - Actually, simpler approach: DON'T modify the watcher. Instead, hook into the git-changed event from the Rust side. In lib.rs setup(), after building tray, register an event listener:
         ```rust
         let app_handle_for_notif = app.handle().clone();
         app.listen("git-changed", move |_event| {
             let state = app_handle_for_notif.state::<Mutex<notifications::NotificationState>>();
             notifications::check_and_notify(&app_handle_for_notif, &state);
             // Also rebuild tray menu since branches may have changed
             let _ = crate::tray::rebuild_tray_menu(&app_handle_for_notif);
         });
         ```
       - This avoids modifying the watcher module and keeps notification logic centralized
       - Import `tauri::Listener` trait in lib.rs for the `listen` method

    Note: Use `std::time::SystemTime::now().duration_since(UNIX_EPOCH).unwrap().as_secs() as i64` for current timestamp comparison with branch.last_commit_timestamp. The branch timestamp is unix seconds (i64).

    1. Create src-tauri/src/fetch.rs with:

       **`start_auto_fetch(app: tauri::AppHandle)`**
       - Spawn a background thread via `std::thread::spawn`
       - In the thread loop:
         a. Load config to read current auto_fetch_interval
         b. If interval == 0, sleep for 60 seconds and re-check (disabled mode — don't spin, just periodically check if user re-enabled)
         c. If interval > 0, enforce minimum of 60 seconds: `let interval = config.settings.auto_fetch_interval.max(60);`
         d. Sleep for `Duration::from_secs(interval as u64)`
         e. Reload config to get current project list (may have changed during sleep)
         f. For each project in config.projects:
            - Call `fetch_remote(&project.path)` — catch errors, log with eprintln, continue to next project
         g. After fetching all projects, emit "git-changed" event so dashboard refreshes:
            ```rust
            let _ = app.emit("git-changed", crate::watcher::GitChangeEvent {
                project_path: "all".to_string(),
                change_type: "fetch_complete".to_string(),
            });
            ```

       **`fetch_remote(project_path: &str) -> Result<(), String>`**
       - First attempt: shell out to `git fetch` via std::process::Command (inherits user's SSH agent and credential helpers — more reliable than git2 for authenticated remotes):
         ```rust
         let output = std::process::Command::new("git")
             .args(["fetch", "--all", "--prune"])
             .current_dir(project_path)
             .output()
             .map_err(|e| format!("git fetch failed: {}", e))?;

         if !output.status.success() {
             let stderr = String::from_utf8_lossy(&output.stderr);
             return Err(format!("git fetch failed: {}", stderr));
         }
         Ok(())
         ```
       - Use `git fetch --all --prune` to fetch all remotes and clean up deleted remote branches
       - Log fetch activity: `eprintln!("[grove] auto-fetch: fetched {}", project_path);`

    2. Update src-tauri/src/lib.rs setup():
       - After the git-changed event listener registration (from Task 1), start auto-fetch:
         ```rust
         let app_handle_fetch = app.handle().clone();
         fetch::start_auto_fetch(app_handle_fetch);
         ```
       - The `mod fetch;` declaration should already exist from Task 1 (add if not)

    Note on credential handling (per RESEARCH.md Pitfall 3): We shell out to `git` CLI instead of using git2::Remote::fetch() because the CLI automatically uses the user's configured SSH agent, credential helpers, and .gitconfig. git2 requires manual credential callback setup which is fragile. The git CLI is guaranteed available since the user is managing git repos.

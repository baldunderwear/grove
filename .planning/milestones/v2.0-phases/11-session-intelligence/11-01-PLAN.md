**1. SessionState enum:**
```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, serde::Serialize)]
#[serde(rename_all = "lowercase")]
pub enum SessionState {
    Working,   // Tool calls streaming, output flowing
    Waiting,   // Claude prompt visible, waiting for user input
    Idle,      // No output for > 60 seconds
    Error,     // Error patterns detected
}
```

**2. ANSI stripper function:**
Strip all ANSI escape sequences from raw PTY bytes before pattern matching. Use a regex approach (per Pitfall 6):
- Strip CSI sequences: `\x1b\[[0-9;]*[a-zA-Z]`
- Strip OSC sequences: `\x1b\].*?\x07` (and `\x1b\].*?\x1b\\`)
- Strip charset selection: `\x1b\(B`, `\x1b\)0`, etc.
- Strip other single-char escapes: `\x1b[=>NH]`
- Use the `regex` crate (add to Cargo.toml if not present). Build the regex once with `std::sync::LazyLock` (Rust 1.80+ stable) -- do NOT compile regex per call.

**3. StateParser struct:**
```rust
pub struct StateParser {
    terminal_id: String,
    current_state: SessionState,
    last_output_time: Instant,
    line_buffer: String,       // Accumulates stripped text until newline
    app_handle: AppHandle,
}
```

Methods:
- `new(terminal_id: String, app_handle: AppHandle) -> Self` -- initial state is `Working` (Claude is starting up)
- `feed(&mut self, raw_data: &str)` -- strip ANSI from raw_data, append to line_buffer, process complete lines, update state
- `check_idle(&mut self)` -- called periodically, transitions to Idle if no output for 60 seconds

**4. Pattern matching for state detection:**
Process each complete line (after ANSI strip) through these patterns:

Waiting patterns (Claude Code prompt visible -- user input expected):
- Line contains the `>` prompt character at start or after whitespace, AND recent output has slowed
- Line matches patterns like: `^\\s*>\\s*$` (bare prompt)
- Line contains "What would you like to do?" or similar Claude prompts
- Line contains the string "claude>" or "> " after a period of output

Working patterns (active output streaming):
- Any non-empty stripped line that doesn't match waiting/error patterns
- Rapid successive lines (output flowing)

Error patterns:
- Line contains "Error:" or "error:" or "ERROR"
- Line contains "panic" or "failed" in error-like context
- Line contains "Permission denied" or "EACCES"
- Only transition to Error if the pattern appears after a period (not mid-stream compiler output)

Idle:
- Not detected from line content -- detected by `check_idle()` timer (60 seconds of no output)

**5. State transition logic:**
- Only emit events on ACTUAL state changes (current != previous)
- Debounce: after transitioning to Working, require 2+ seconds of silence before transitioning to Waiting (prevents flicker during rapid tool output)
- Emit via Tauri events (low frequency, fine for events): `app_handle.emit("session-state-changed", payload)` where payload is:
```rust
#[derive(Clone, serde::Serialize)]
pub struct SessionStatePayload {
    pub terminal_id: String,
    pub state: SessionState,
    pub timestamp: u64,  // Unix timestamp millis
}
```

**3. Update the PTY reader thread** (lines 93-120) to feed data to the state parser:
The key insight: the reader thread already converts raw bytes to a String for the Channel. We feed that same string to the parser AFTER sending to xterm.js (latency-free rendering):

```rust
std::thread::Builder::new()
    .name(format!("pty-reader-{}", &id_for_thread[..8]))
    .spawn(move || {
        let mut buf = [0u8; 4096];
        loop {
            match reader.read(&mut buf) {
                Ok(0) => {
                    let _ = on_event.send(TerminalEvent::Exit { code: None });
                    break;
                }
                Ok(n) => {
                    let data = String::from_utf8_lossy(&buf[..n]).to_string();
                    // Stream to xterm.js FIRST (no latency impact)
                    if on_event.send(TerminalEvent::Data { data: data.clone() }).is_err() {
                        break;
                    }
                    // Feed to state parser (async-ish, in same thread but after send)
                    parser.feed(&data);
                }
                Err(e) => {
                    let _ = on_event.send(TerminalEvent::Error {
                        message: format!("PTY read error: {}", e),
                    });
                    break;
                }
            }
        }
    })
```

**4. Add idle check timer** inside the reader thread. Every 10 seconds between reads, call `parser.check_idle()`. Use a non-blocking approach: set a read timeout on the reader if possible, or use a separate lightweight timer thread that sends a check signal. Simplest approach: after each successful read, check if enough time has elapsed since last idle check and call `check_idle()` if so. The idle timer doesn't need to be precise -- checking every read cycle is sufficient since if there's no output, the read blocks and idle is detected on the next read return.

Actually, better approach: spawn a tiny companion timer thread alongside the reader:
```rust
let parser_idle = Arc::new(Mutex::new(/* reference to parser or just a flag */));
```

Wait -- simplest correct approach: the parser tracks `last_output_time` internally. When the NEXT read arrives after a gap, `feed()` checks elapsed time and emits idle if >60s. This means idle detection has slight lag (triggered on next output after silence) but is correct and simple. For truly prompt idle detection (no more output at all), add a timer thread:

```rust
// Idle detection timer thread
let idle_terminal_id = terminal_id.clone();
let idle_app_handle = app_handle.clone(); // clone before moving into reader
std::thread::Builder::new()
    .name(format!("pty-idle-{}", &id_for_thread[..8]))
    .spawn(move || {
        // Simple approach: periodically check if session still exists
        // and emit idle if no output for 60s. Use a shared AtomicU64 for last_output_time.
        loop {
            std::thread::sleep(std::time::Duration::from_secs(15));
            // Check via shared atomic -- if reader thread exited, this will just keep checking harmlessly
            // The TerminalManager::kill() cleanup will eventually drop things
        }
    });
```

Actually, the cleanest approach: use a `std::sync::Arc<std::sync::atomic::AtomicU64>` shared between the reader thread and an idle-check timer thread. The reader updates the timestamp on each read. The idle timer checks every 15 seconds and emits idle event if elapsed > 60s. When the reader thread exits, set a flag so the idle thread also exits.

Implement this with:
- `Arc<AtomicU64>` for `last_output_epoch_ms` (shared between reader + idle threads)
- `Arc<AtomicBool>` for `reader_alive` (set false when reader exits, idle thread checks and exits)

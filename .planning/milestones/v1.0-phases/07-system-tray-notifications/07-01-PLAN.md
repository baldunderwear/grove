  <read_first>

  <action>
    1. Create src-tauri/src/tray.rs with two public functions:

       **`build_tray(app: &mut tauri::App) -> Result<(), Box<dyn std::error::Error>>`**
       - Move ALL tray-building code from lib.rs setup() into this function
       - Use TrayIconBuilder::with_id("grove-tray") as before
       - Set show_menu_on_left_click(false) as before
       - Register ONE on_menu_event handler that pattern-matches ALL menu item IDs:
         - "quit" => app.exit(0)
         - "show" => show/focus main window
         - "settings" => emit "navigate" event with payload "settings" to frontend
         - IDs starting with "wt-" => extract index, look up worktree from config, emit "launch-worktree" event with worktree path
         - _ => no-op
       - Keep the existing on_tray_icon_event left-click toggle behavior (FR-05.4)
       - Call rebuild_tray_menu() at end to set initial menu
       - Also move the window close-intercept code (on_window_event/prevent_close/hide) into this function

       **`rebuild_tray_menu(app: &tauri::AppHandle) -> Result<(), Box<dyn std::error::Error>>`**
       - Retrieve tray via `app.tray_by_id("grove-tray")`
       - Load config via `crate::config::persistence::load_or_create_config(app)`
       - Get selected project (first project if any)
       - If a project exists, call `crate::git::branches::list_worktree_branches()` to get branches
       - Filter to branches with active worktree paths, take up to 10 most recent (by last_commit_timestamp descending)
       - Build menu structure:
         ```
         Open Grove          (id: "show")
         ─────────────────
         Recent Worktrees >  (submenu)
           wt/feature-a      (id: "wt-0")
           wt/feature-b      (id: "wt-1")
           ...
         Settings             (id: "settings")
         ─────────────────
         Quit Grove           (id: "quit")
         ```
       - If no projects registered, skip the Recent Worktrees submenu
       - Use `tray.set_menu(Some(menu))` to update
       - Import: `use tauri::menu::{Menu, MenuItem, PredefinedMenuItem, SubmenuBuilder};`
       - Import: `use tauri::tray::{MouseButton, MouseButtonState, TrayIconBuilder, TrayIconEvent};`
       - Import: `use tauri::{Emitter, Manager};`

    2. Update src-tauri/src/lib.rs:
       - Add `mod tray;` declaration after existing mod declarations
       - Register notification and autostart plugins in the Builder chain:
         ```rust
         .plugin(tauri_plugin_notification::init())
         .plugin(tauri_plugin_autostart::init(
             tauri_plugin_autostart::MacosLauncher::LaunchAgent,
             None,
         ))
         ```
       - Replace ALL tray-building code in setup() with: `tray::build_tray(app)?;`
       - Remove the window close-intercept code (moved to tray.rs)
       - Keep the file watcher startup code as-is
       - Remove unused tray-related imports from lib.rs (Menu, MenuItem, PredefinedMenuItem, TrayIconBuilder, TrayIconEvent, MouseButton, MouseButtonState)

    3. Add a Tauri command for triggering tray rebuild from frontend (called after config changes):
       In tray.rs, add:
       ```rust
       #[tauri::command]
       pub fn refresh_tray(app_handle: tauri::AppHandle) -> Result<(), String> {
           rebuild_tray_menu(&app_handle).map_err(|e| e.to_string())
       }
       ```
       Register in lib.rs invoke_handler: `commands::session_commands::..., tray::refresh_tray,`

    Note: The on_menu_event handler is registered ONCE during build_tray() on the TrayIcon itself. It persists across set_menu() calls (per Tauri 2 API). Dynamic menu items added later via set_menu() will have their events routed to this same handler. Use ID prefix pattern-matching (e.g., starts_with("wt-")) for dynamic items.
